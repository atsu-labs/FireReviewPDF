# Git開発ワークフロー・スキル

本スキルは、AIエージェントが安全かつ規約に準拠した形で、ブランチ作成からプルリクエスト（PR）の作成、さらにはPRに対するレビューコメントの確認と修正・返答までの一連のGit開発フローを自動実行するための指示書である。

## 0. 基本ルール（共通）
- **日本語の徹底**: コミットメッセージ、プルリクエストのタイトル・説明文、レビュー返信など、ユーザーや他の開発者とのテキストコミュニケーションは必ず**日本語**で記述する。
- **指示書の参照**: コミットメッセージやPRの記述内容、コード規約の詳細については、必ず事前に [development-rules.md](../../rules/development-rules.md) を読み込み、その指示を厳格に遵守すること。

---

## 1. 開発開始時のフロー (ブランチ作成)
新機能の開発やバグ修正を開始する際は、以下の手順を実行する。

1. **ベースブランチの同期**:
   - `git checkout main` でメインブランチへ切り替える。
   - `git pull origin main` を実行し、リモートの最新コードをローカルに完全同期する。
2. **新規機能ブランチの作成**:
   - 最新の `main` ブランチから機能ブランチを作成する。
   - ブランチ名の命名規則は、タスクの性質に応じて以下のように命名する：
     - 新機能: `feat/feature-name` (例: `feat/pdf-preview`)
     - バグ修正: `fix/bug-name` (例: `fix/calibration-bug`)
     - リファクタリング: `refactor/refactor-name` (例: `refactor/canvas-tool`)
     - ドキュメント修正: `docs/docs-name`
   - コマンド: `git checkout -b <branch-name>`

---

## 2. 開発完了後のフロー (検証・コミット・プッシュ)
コードの変更が完了した後は、以下の手順で変更を検証し、リポジトリへ反映する。

1. **変更の検証**:
   - 変更されたコードが動作することを確認するため、テストを実行する（例: `python src/main.py` 等での動作確認やユニットテストの実行）。
   - コードが正常にビルド・動作し、エラーが出ないことを検証してからコミットへ進む。
2. **コミットメッセージの作成**:
   - `development-rules.md` に基づき、日本語でコミットメッセージを作成する。
   - 1行目（件名）は50文字以内で変更内容を簡潔に要約する。
   - 形式: `feat: XXXの機能を追加` または `fix: XXXのバグを修正` など。
   - コマンド:
     - `git add <対象ファイル>`
     - `git commit -m "feat: 件名" -m "本文（必要に応じて詳細）"`
3. **プッシュ**:
   - 作成した機能ブランチをリモートへプッシュする。
   - コマンド: `git push origin <branch-name>`

---

## 3. プルリクエスト (PR) の作成フロー
プッシュ完了後、以下の手順でプルリクエストを作成する。

1. **作成手段の確認**:
   - 環境内で GitHub CLI (`gh` コマンド) が使用可能か確認する。
2. **GitHub CLI による作成 (優先)**:
   - `gh` が使える場合、以下のテンプレート項目を含む説明文を日本語で作成し、コマンドラインからPRを作成する。
     - **変更概要**: 何を変更したか
     - **変更理由**: なぜ変更が必要だったか
     - **テスト方法**: 動作確認の手順
     - **関連 Issue**: 対応する Issue 番号（例: `Closes #123`）
   - コマンド例:
     ```bash
     gh pr create --title "feat: PDF プレビュー機能を追加" --body "## 変更概要\n...\n\n## 変更理由\n...\n\n## テスト方法\n...\n\n## 関連 Issue\nCloses #123"
     ```
3. **Markdownファイルによる代替作成 (ghが使えない場合)**:
   - GitHub CLIが利用できない、または権限エラーが発生した場合は、PRのタイトルと上記テンプレートに沿った説明文を記載したマークダウンファイルをプロジェクトルートに生成し、ユーザーに手動でPRを作成するよう提示する。
   - ファイル名: `PR_TEMPLATE_FOR_USER.md`
   - ファイル作成後、ユーザーにチャットでその旨を報告する。

---

## 4. PRレビュー指摘への対応フロー
作成したPRに対し、レビュアー（ユーザー）から指摘が入った場合は、以下の手順で修正および返答を行う。

1. **レビューコメントの取得**:
   - `gh pr view --comments` または `gh api` を使用して、現在のPRに寄せられたコメントやレビューの内容を取得・確認する。
   - コマンドが使えない場合は、ユーザーからのチャット経由の指摘内容をベースにする。
2. **コードの修正と再検証**:
   - 指摘された内容に従ってコードを修正する。
   - 修正後、再度アプリケーションを起動するなどして動作検証を行う。
3. **修正内容のコミットとプッシュ**:
   - `git add` および適切な日本語のコミットメッセージでコミットし、リモートブランチへプッシュする。
4. **レビューコメントへの返信**:
   - 修正が完了したことをレビューコメントの返答スレッド、またはPRの全体コメントに書き込む。
   - **重要ルール**: `development-rules.md` に従い、「レビューありがとうございます」や「お世話になります」といった**挨拶や定型的な表現は一切含めない**こと。技術的な対応内容、設計判断、結論のみを簡潔に記述する。
   - コマンド例:
     ```bash
     gh pr comment <PR番号> --body "〇〇の指摘について、XXXの処理をYYYに変更することで対応しました。"
     ```
